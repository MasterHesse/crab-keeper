# Crab-Keeper 项目设计文档

## 1. 项目概述

Crab-Keeper 是一个使用 Rust 语言实现的分布式协调服务，其设计灵感源自 Apache ZooKeeper。项目兼具两个目标：其一，实践 Rust 系统编程；其二，以 ITMO 大学分布式系统课程的理论体系为指引，通过工程实现将其内化。

项目最终交付形态为一个可独立部署的服务，提供 TUI 或 GUI 交互界面（占用本地端口），供他人测试与使用。

## 2. 核心概念与运作逻辑

### 2.1 分布式通信模型

系统建立在"通过消息传递进行通信"的核心模型之上。各进程（节点）之间不共享内存，仅通过消息收发实现同步与协调。每个进程在启动（STARTED）和结束（DONE）时需完成同步，确保整体状态一致。

### 2.2 时序与事件排序

系统不依赖物理时钟，而是通过 Lamport 逻辑时钟算法为每个事件（发送/接收）分配逻辑时间戳。逻辑时间戳决定了分布式环境中事件的全局偏序关系，是解决状态不一致问题的理论基础。

### 2.3 关键算法路径

项目严格遵循 ITMO 课程的四阶段递进路径：

- **阶段一 - 通信框架**：实现进程间的消息收发与同步（parent_work / child_work）。
- **阶段二 - 分布式业务**：在通信基础之上引入具体业务逻辑（如分布式银行系统的跨进程转账），关注事务原子性与一致性。
- **阶段三 - 逻辑时钟**：实现 Lamport 逻辑时钟，移除对物理时钟的依赖，以逻辑时间修正状态不一致。
- **阶段四 - 分布式互斥**：实现 Ricart-Agrawala 算法，解决多进程对临界区（Critical Section）的安全竞争访问。

### 2.4 协调服务能力

在课程算法基础之上，向 ZooKeeper 能力对齐：

- **配置管理**：集中存储配置数据，支持动态分发与热更新。
- **命名服务**：提供服务注册与发现，使消费者能动态定位提供者。
- **分布式同步**：提供分布式锁等协调原语，管理共享资源访问。
- **集群管理**：追踪集群节点状态，支持领导者选举。

## 4. 数据模型

### 4.1 分层命名空间 (Znode Tree)

数据模型采用类似文件系统的分层树形结构，根节点为 `/`，每个数据节点称为 Znode。Znode 既可存储数据（如配置值），也可拥有子节点，形成任意深度的层级命名空间。

### 4.2 临时节点 (Ephemeral Nodes)

临时节点的生命周期与创建它的客户端会话绑定。当会话终止（主动断开或超时），对应节点自动删除。此机制是实现服务发现与集群成员管理的核心——服务实例注册为临时节点后，宕机即自动摘除。

### 4.3 顺序节点 (Sequential Nodes)

创建节点时可指定顺序模式，系统自动在节点名称尾部追加单调递增序号。该机制是实现分布式锁与分布式队列的基础——多个竞争者通过创建顺序节点来排队获取锁，序号最小者获得锁。

### 4.4 监视器 (Watcher)

客户端可在 Znode 上注册监视回调。当节点数据或子节点列表发生变化时，服务端主动向客户端推送通知。此机制是配置动态更新与服务发现实时性的核心保障。

## 5. 模块划分

| 模块 | 职责 | 对应课程阶段 |
|------|------|-------------|
| `communication` | 进程间消息收发与同步机制 | Lab 1 |

| `mutex` | Ricart-Agrawala 分布式互斥 | Lab 4 |

| `session` | 客户端会话管理与临时节点绑定 | ZooKeeper 核心 |
| `election` | 领导者选举 | ZooKeeper 集群 |
| `server` | 服务端整合与对外接口 | 集成 |

## 6. 技术约束

- **语言**：以 Rust 为主要开发语言，必要时可搭配脚本语言辅助。
- **开发模式**：严格遵循 TDD（测试驱动开发），每阶段先编写测试用例定义输入输出，再进行实现。
- **构建**：基于 Cargo 构建系统。
- **CI/CD**：使用 GitHub Actions 进行持续集成与交付。
- **版本管理**：使用 Git 进行分支管理与版本迭代。
- **运行环境**：单机本地环境，WSL Ubuntu。
- **部署形态**：编译为独立可执行文件，通过本地端口暴露 TUI/GUI 界面。
