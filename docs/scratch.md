# 项目设想 crab-keeper
> 此文件为项目的设想，作为项目设计文件和项目进度管理文件的生成参考

## 项目描述
项目名称 “crab-keeper” 源自于 ZooKeeper，但是使用 Rust 语言进行实现。项目主要的目的有两个：
1. 使用 Rust 语言进行开发和设计。
2. 学习 ITMO 大学的分布式系统课程，并且实现一个类似 Zookeeper 的项目来付诸实践。

## 项目开发环境
1. 个人开发项目
2. Windows 的 WSL Ubuntu 环境
3. 使用 Rust 语言进行开发
4. 使用 Vscode 进行开发
5. 单机环境，没有云服务器

## 项目基本要求
排名不分先后，皆为基础需求，必须贯彻每一个阶段：
1. 使用 Github Action 进行 CICD；
2. 使用 Git 进行版本管理；
3. 主要 Rust 开发，必要时可以使用其他脚本语言；
4. 使用 TDD （测试驱动） 的开发模式，每阶段进行时，先写好测试的输入和输出，再进行实际开发；
5. 使用模块化的开发模式，尽可能的以功能划分模块；
6. 最终交付应该是一个成品服务，可以被他人部署，测试和把玩

## 项目进度安排

### ITMO 课程的四个学习阶段
- Lab 1: 初识分布式通信 (Introduction to communication framework) - 奠定基石。这个实验的目标是熟悉课程提供的通信框架 (libdistributedmodel.so)，实现进程间的消息收发与同步。学生需要实现parent_work()和child_work()函数，让所有子进程在启动（STARTED）和结束（DONE）时进行同步。这让学生直观地理解分布式系统中“通过消息传递进行通信”的核心模型。
- Lab 2: 实现分布式银行系统 (Distributed banking system) - 引入业务逻辑。在Lab 1的基础上，这个实验引入了具体的业务场景：一个由父进程（协调者）和多个子进程（银行分支）组成的分布式银行系统。学生需要实现跨进程的资金转账功能，并开始关注分布式系统中的核心问题——一致性（如转账的原子性）和物理时间（如使用get_physical_time()）。
- Lab 3: Lamport逻辑时钟 (Lamport's logical clocks) - 解决时间问题。这个实验是课程的关键转折点。它要求移除对理想物理时钟的依赖，转而由学生自己实现Lamport的逻辑时钟算法。学生需要为每个事件（发送/接收）分配逻辑时间戳，并用它来修正上一实验中可能出现的状态不一致问题。这让学生深刻理解在分布式系统中，如何用逻辑时间替代物理时间来定义事件顺序。
- Lab 4: 分布式互斥算法 (Distributed mutual exclusion algorithm) - 登顶之作。作为课程的压轴实验，它要求学生在已有基础上，实现 Ricart-Agrawala 分布式互斥算法。在这个实验中，多个子进程需要竞争进入临界区（Critical Section） 以安全地调用print()函数。学生必须综合运用前三个实验的所有知识：进程通信（Lab 1）、分布式协调（Lab 2）、以及 Lamport时钟（Lab 3）来解决分布式环境下的资源竞争问题。

### ZooKeeper 的核心能力

- 配置管理：集中管理和动态分发配置。
- 命名服务：提供服务注册与发现，让服务消费者能动态找到提供者。
- 分布式同步：提供分布式锁等机制，协调对共享资源的访问。
- 集群管理：管理集群节点状态，进行领导者选举等

### ZooKeeper 的核心组件（关键设计）
- 数据模型 (Data Model)：一个类似文件系统的分层命名空间，其中的数据节点称为 Znode。Znode 可以存储数据（如配置信息），也可以有子节点。
- 临时节点 (Ephemeral Nodes)：一种特殊的 Znode，其生命周期与创建它的客户端会话绑定。会话结束，节点自动删除。这是实现服务发现和集群管理的核心。
- 监视器 (Watcher)：客户端可以在 Znode 上设置“监视”。当节点数据或子节点发生变化时，ZooKeeper 会通知客户端。这是实现配置动态更新和服务发现的核心。
- 顺序节点 (Sequential Nodes)：创建节点时，ZooKeeper 会自动在节点名后附加一个单调递增的序号。这是实现分布式锁和队列等高级同步功能的基础。
- 高可用架构：ZooKeeper 自身以集群方式部署，通过 Zab 协议（ZooKeeper Atomic Broadcast）保持数据一致性。只要集群中超过半数的节点正常工作，服务就可用
